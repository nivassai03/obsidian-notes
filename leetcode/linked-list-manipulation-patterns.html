Linked List Manipulation Patterns

Merge Sorted Lists

Merge Two Sorted Lists

Intitution:

  • We are given two already sorted linked lists
  • Goal is to merge them into one sorted list without creating new nodes
  • Since both lists are sorted, smallest element is always at the head of one of the lists
  • So at every step, we pick the smaller node and attach it to result
  • This is same idea as merge step in merge sort
  • Use a dummy node to simplify handling of head

Key observation:

  • Only need to compare current nodes of both lists
  • Once one list is exhausted, remaining part of other list is already sorted
  • No need for extra space, we can reuse existing nodes

Approach:

  • Create a dummy node and a tail pointer
  • Iterate while both lists are non-empty:
    • Compare l1 and l2 values
    • Attach smaller node to tail
    • Move that list pointer forward
    • Move tail forward
  • After loop, attach remaining nodes of non-empty list
  • Return dummy->next

Invariant:

  • At any point, result list is sorted
  • tail always points to last node of merged list
  • l1 and l2 always point to next candidates to compare

Edge cases (if any):

  • One list is empty → return the other
  • Both lists empty → return NULL
  • Duplicate values → handled naturally

Time & Space complexity:

  • Time: O(n + m)
  • Space: O(1)
class Solution {
public:
    ListNode* mergeTwoLists(ListNode* l1, ListNode* l2) {
        ListNode dummy(0);
        ListNode* tail = &dummy;
        while (l1 && l2) {
            if (l1->val <= l2->val) {
                tail->next = l1;
                l1 = l1->next;
            } else {
                tail->next = l2;
                l2 = l2->next;
            }
            tail = tail->next;
        }
        // attach remaining
        tail->next = l1 ? l1 : l2;
        return dummy.next;
    }
};

Merge K Sorted Lists

Intitution:

  • We need to merge k sorted linked lists into one sorted linked list.
  • For merging 2 sorted lists, we already know the standard merge process:
    • compare heads
    • attach smaller node
    • move that list forward
  • The real question is not how to merge 2 lists, but how to organize merging of k lists efficiently.
  • A natural idea is:
    • merge list 0 with list 1
    • then merge result with list 2
    • then merge result with list 3
  • This works, but it is slow because the merged list keeps becoming larger, so later merges cost more.
  • Better idea:
    • merge lists in pairs, like merge sort
    • first merge nearby pairs
    • then merge the merged results
    • keep doing this until only one list remains
  • This keeps merges balanced, which is why it becomes efficient.

Key observation:

  • Merging 2 sorted linked lists takes time proportional to total nodes in those 2 lists.
  • So we should avoid making one huge merged list too early.
  • Pairwise balanced merging makes each node participate in only about log k merge levels.
  • This gives O(N log k) instead of O(Nk).

Approach:

  • Let lists be the array of linked list heads.
  • Use interval = 1
  • interval tells us how far apart the two lists are that should be merged in current round.
  • Initially interval = 1, so:
    • merge lists[0] with lists[1]
    • merge lists[2] with lists[3]
    • merge lists[4] with lists[5]
  • Store each merged result back at the first position:
    • lists[0] = merge(lists[0], lists[1])
    • lists[2] = merge(lists[2], lists[3])
  • After finishing one round, double the interval:
    • interval = interval * 2
  • Now interval = 2, so:
    • merge lists[0] with lists[2]
    • merge lists[4] with lists[6]
  • Again double the interval.
  • Keep doing this until interval >= k
  • Final answer will be in lists[0]

Invariant:

  • Before each round with some interval, every lists[i] at positions 0, 2*interval, 4*interval, ... stores a merged sorted list formed from a block of original lists.
  • After merging lists[i] with lists[i + interval], lists[i] becomes the merged sorted list for a larger block.
  • At every step, all stored lists remain sorted.

Edge cases (if any):

  • lists is empty → return nullptr
  • Some lists are empty → merge function handles them naturally
  • Only one list present → return it directly
  • Number of lists is odd → one list in that round may remain unmerged, and it will be merged in a later round

Time & Space complexity:

  • Time: O(N log k), where N is total number of nodes across all lists
  • Space: O(1) extra if iterative merging is used, excluding input storage

Interval intuition in detail
Intitution:

  • interval is the gap between the two lists we merge in current round.
  • It starts at 1 because initially we merge adjacent lists.
  • After one round, each merged list now represents 2 original lists.
  • So next time we should merge results that are 2 positions apart.
  • Then each merged list represents 4 original lists, so next gap becomes 4.
  • That is why interval keeps doubling:
    • 1, 2, 4, 8, ...

Key observation:

  • interval is not about node distance inside a linked list.
  • It is about index distance inside the lists array.
  • It tells us which two already-merged groups should now be combined.

Approach:

  • Suppose:

    • lists = [l0, l1, l2, l3, l4, l5, l6, l7]

  • Round 1:

    • interval = 1
    • merge:
      • lists[0] with lists[1]
      • lists[2] with lists[3]
      • lists[4] with lists[5]
      • lists[6] with lists[7]
    • now positions 0, 2, 4, 6 hold merged lists of size 2 groups

  • Round 2:

    • interval = 2
    • merge:
      • lists[0] with lists[2]
      • lists[4] with lists[6]
    • now positions 0, 4 hold merged lists of size 4 groups

  • Round 3:

    • interval = 4
    • merge:
      • lists[0] with lists[4]
    • now lists[0] holds all 8 lists merged

Invariant:

  • At start of round with interval = x, each valid position stores the merged result of x original lists.
  • After that round, each valid position stores the merged result of 2x original lists.

Why i += interval * 2

Intitution:

  • In one round, we merge pairs of groups.
  • Each group has size interval.
  • So one full pair covers 2 * interval positions.
  • After processing one pair, next pair starts after those 2 * interval positions.

Key observation:

  • If interval = 1, pairs are:
    • (0,1), (2,3), (4,5)
    • so i moves by 2
  • If interval = 2, pairs are:
    • (0,2), (4,6)
    • so i moves by 4
  • If interval = 4, pairs are:
    • (0,4), (8,12)
    • so i moves by 8

Approach:

  • We write:
    • for (int i = 0; i + interval < n; i += interval * 2)
  • i + interval < n checks that partner list exists
  • i += interval * 2 jumps to next pair block

Invariant:

  • In one round, every pair of mergeable groups is processed exactly once.
  • No overlap happens between groups in same round.

Code

class Solution {
private:
    ListNode* merge(ListNode* l1, ListNode* l2) {
        ListNode dummy(0);
        ListNode* tail = &dummy;

        while (l1 && l2) {
            if (l1->val <= l2->val) {
                tail->next = l1;
                l1 = l1->next;
            } else {
                tail->next = l2;
                l2 = l2->next;
            }
            tail = tail->next;
        }
        tail->next = l1 ? l1 : l2;
        return dummy.next;
    }

public:
    ListNode* mergeKLists(vector<ListNode*>& lists) {
        if (lists.empty()) return nullptr;

        int n = lists.size();
        int interval = 1;

        while (interval < n) {
            for (int i = 0; i + interval < n; i += interval * 2) {
                lists[i] = merge(lists[i], lists[i + interval]);
            }
            interval *= 2;
        }

        return lists[0];
    }
};

In-place Reversal

Reverse Linked List

Intitution:

  • We need to reverse direction of every pointer in the linked list
  • Instead of changing values, we change links between nodes
  • At any moment, we maintain:
    • one part already reversed
    • one part still to process
  • Build reversed list step by step from left to right

Key observation:

  • A node only needs to point to its previous node
  • But before breaking next, we must store it
  • So we always need 3 pointers:
    • prev → reversed part
    • curr → current node
    • next → to not lose remaining list

Approach:

  • Initialize:
    • prev = nullptr
    • curr = head
  • While curr is not null:
    • store next node → next = curr->next
    • reverse link → curr->next = prev
    • move prev → prev = curr
    • move curr → curr = next
  • At end:
    • prev becomes new head

Invariant:

  • prev always points to fully reversed portion
  • curr always points to next node to process
  • nodes before prev are correctly reversed
  • nodes after curr are untouched

Edge cases (if any):

  • empty list → return nullptr
  • single node → remains same
  • two nodes → simple swap

Time & Space complexity:

  • Time: O(n) (visit each node once)
  • Space: O(1) (in-place reversal)
class Solution {
public:
    ListNode* reverseList(ListNode* head) {
        ListNode* prev = nullptr;
        ListNode* curr = head;
        while (curr != nullptr) {
            ListNode* nextNode = curr->next;
            curr->next = prev;
            prev = curr;
            curr = nextNode;
        }
        return prev;
    }
};

Remove Duplicates From Sorted List

Intitution:

  • Given a sorted linked list, duplicates will always appear next to each other
  • We don’t need extra memory or hashing
  • Goal is to remove duplicate nodes but keep one occurrence
  • Instead of tracking values separately, we can directly compare adjacent nodes

Key observation:

  • If current node value == next node value → next node is duplicate
  • We should remove the next node, not the current one
  • Stay at same node after deletion because more duplicates may exist

Approach:

  • Start with pointer cur = head
  • Traverse while cur and cur->next exist
  • If cur->val == cur->next->val:
    • store duplicate node
    • skip it → cur->next = cur->next->next
    • delete duplicate node
  • Else:
    • move cur forward
  • Return head

Invariant:

  • All nodes before cur are already processed and contain no duplicates
  • cur always points to the last unique node in processed part
  • Remaining list (after cur) is yet to be processed

Edge cases (if any):

  • empty list → return nullptr
  • single node → return as is
  • all nodes same → only one node remains
  • no duplicates → list unchanged

Time & Space complexity:

  • Time: O(n) (each node processed once)
  • Space: O(1) (in-place modification)
class Solution {
public:
    ListNode* deleteDuplicates(ListNode* head) {
        ListNode* cur = head;

        while (cur != nullptr && cur->next != nullptr) {
            if (cur->val == cur->next->val) {
                ListNode* temp = cur->next;
                cur->next = cur->next->next;
                delete temp;
            } else {
                cur = cur->next;
            }
        }

        return head;
    }
};

Palindrome Linked List

Intitution:

  • A palindrome means the sequence reads same forward and backward
  • In arrays, we use two pointers from both ends
  • In linked list, we cannot move backward
  • So we transform the problem:
    • find middle of list
    • reverse second half
    • compare both halves in forward direction
  • This simulates two-pointer comparison

Key observation:

  • Using slow & fast pointers, we can find middle in one pass
  • Reversing second half allows forward comparison
  • For odd length, middle element does not affect palindrome → skip it

Approach:

  • Handle edge cases (0 or 1 node → true)
  • Use slow and fast pointers to find middle
  • If list length is odd → move slow one step ahead (skip middle)
  • Reverse list starting from slow
  • Compare:
    • first pointer from head
    • second pointer from reversed half
  • If all values match → palindrome
  • (Optional) reverse second half again to restore list

Invariant:

  • slow always moves 1 step, fast moves 2 steps → slow reaches middle
  • reversed second half always mirrors first half if palindrome
  • during comparison:
    • first pointer scans first half
    • second pointer scans reversed half
    • all compared nodes must match

Edge cases (if any):

  • empty list → true
  • single node → true
  • even length → split into equal halves
  • odd length → ignore middle element
  • all same values → always palindrome

Time & Space complexity:

  • Time: O(n) (middle + reverse + compare)
  • Space: O(1) (in-place operations)

Code

class Solution {
public:
    ListNode* reverseList(ListNode* head) {
        ListNode* prev = nullptr;
        ListNode* curr = head;

        while (curr != nullptr) {
            ListNode* nextNode = curr->next;
            curr->next = prev;
            prev = curr;
            curr = nextNode;
        }
        return prev;
    }

    bool isPalindrome(ListNode* head) {
        if (head == nullptr || head->next == nullptr) return true;
        ListNode* slow = head;
        ListNode* fast = head;

        while (fast != nullptr && fast->next != nullptr) {
            slow = slow->next;
            fast = fast->next->next;
        }
        if (fast != nullptr) slow = slow->next;
        ListNode* secondHead = reverseList(slow);
        ListNode* copySecond = secondHead;
        ListNode* first = head;

        bool isPal = true;
        while (secondHead != nullptr) {
            if (first->val != secondHead->val) {
                isPal = false;
                break;
            }
            first = first->next;
            secondHead = secondHead->next;
        }
        reverseList(copySecond); // restore
        return isPal;
    }
};

Remove Duplicates from Sorted List II

Intitution:

  • Given a sorted linked list, duplicates appear in contiguous blocks
  • Unlike simpler version, we must remove all nodes that have duplicates
  • Only nodes that appear exactly once should remain
  • So instead of removing individual duplicates, we remove entire blocks of repeated values

Key observation:

  • If a value appears more than once, none of those nodes should remain
  • We need to detect the start of a duplicate block and skip all nodes of that value
  • A dummy node is required to handle cases where head itself gets deleted

Approach:

  • Create dummy node pointing to head
  • Use two pointers:
    • prev → last node confirmed to be unique
    • cur → current node being checked
  • Traverse list:
    • If cur starts a duplicate block (cur->val == cur->next->val):
      • store duplicate value
      • skip all nodes with this value
      • connect prev->next to next non-duplicate node
    • Else:
      • move prev forward (this node is unique)
  • Return dummy.next

Invariant:

  • prev always points to last valid (unique) node in result list
  • cur scans the list and identifies duplicate blocks
  • nodes before prev are guaranteed to have no duplicates
  • nodes skipped are completely removed if duplicate

Edge cases (if any):

  • empty list → return nullptr
  • single node → return as is
  • all nodes duplicated → result is empty list
  • duplicates at head → handled using dummy node
  • duplicates at end → properly skipped

Time & Space complexity:

  • Time: O(n) (each node visited once)
  • Space: O(1) (in-place modification)
class Solution {
public:
    ListNode* deleteDuplicates(ListNode* head) {
        ListNode dummy(0, head);
        ListNode* prev = &dummy;
        ListNode* cur = head;

        while (cur != nullptr) {
            // detect duplicate block
            if (cur->next != nullptr && cur->val == cur->next->val) {
                int dupVal = cur->val;

                // skip all nodes with dupVal
                while (cur != nullptr && cur->val == dupVal) {
                    ListNode* temp = cur;
                    cur = cur->next;
                    delete temp;
                }
                prev->next = cur;
            } else {
                prev = cur;
                cur = cur->next;
            }
        }
        return dummy.next;
    }
};

Reverse Linked List II

Intitution:

  • We need to reverse only a portion of the linked list (from position left to right)
  • Full reversal logic cannot be directly applied because:
    • nodes before left and after right must remain unchanged
  • Key idea:
    • keep the part before left fixed
    • reverse the sublist in-place
    • reconnect everything correctly
  • Instead of classic reversal, we use head insertion:
    • repeatedly take next node and move it to the front of the sublist
  • This avoids breaking connections and keeps reversal local

Key observation:

  • We can treat the sublist independently if we know:
    • node before sublist (prev)
    • first node of sublist (curr)
  • Reversal can be done by:
    • removing node after curr
    • inserting it right after prev
  • Repeat (right - left) times to reverse full sublist

Approach:

  • Create dummy node → handles case when left = 1
  • Move prev pointer to node before position left
  • Set curr = prev->next (start of sublist)
  • Loop (right - left) times:
    • temp = curr->next
    • detach temp → curr->next = temp->next
    • insert temp after prev:
      • temp->next = prev->next
      • prev->next = temp
  • Return dummy.next

Invariant:

  • prev always points to node before reversed sublist
  • curr always points to tail of reversed portion
  • nodes between prev and curr are already reversed
  • nodes after curr are yet to be processed
  • list remains connected at all times (no broken chains)

Edge cases (if any):

  • empty list → return nullptr
  • single node → no change
  • left == right → no reversal needed
  • left = 1 → handled by dummy node
  • reversing entire list → behaves like full reverse

Time & Space complexity:

  • Time: O(n) (single traversal + partial reversal)
  • Space: O(1) (in-place pointer manipulation)

Code

class Solution {
public:
    ListNode* reverseBetween(ListNode* head, int left, int right) {
        if (!head || left == right) return head;
        ListNode dummy(0);
        dummy.next = head;
        ListNode* prev = &dummy;
        // 1. Move prev to node before 'left'
        for (int i = 1; i < left; i++) {
            prev = prev->next;
        }
        // 2. Start reversing
        ListNode* curr = prev->next;
        for (int i = 0; i < right - left; i++) {
            ListNode* temp = curr->next;
            curr->next = temp->next;
            temp->next = prev->next;
            prev->next = temp;
        }
        return dummy.next;
    }
};

Intersection Detection

Intersection of Two Linked Lists

Intitution:

  • We need to find the first common node where two linked lists intersect
  • Intersection means same node in memory, not same value
  • After intersection point, both lists share the same tail
  • Problem reduces to aligning both lists so we can compare nodes one by one

Key observation:

  • If we traverse both lists fully, lengths may differ
  • But if we switch heads after reaching end, both pointers travel equal distance (m + n)
  • This cancels out length difference automatically
  • Eventually both pointers either meet at intersection or both reach null

Approach:

  • Use two pointers a and b
  • Start a at headA and b at headB
  • Move both one step at a time
  • If a reaches null → move it to headB
  • If b reaches null → move it to headA
  • Continue until a == b
  • Return a (intersection node or null)

Invariant:

  • Both pointers traverse exactly same total distance (m + n)
  • After switching, distance difference is neutralized
  • If intersection exists, both pointers will align at intersection
  • If no intersection, both pointers will become null at same time

Edge cases (if any):

  • One or both lists are empty → return null
  • No intersection → both pointers end at null
  • Intersection at head → immediate match
  • Different lengths → handled by pointer switching

Time & Space complexity:

  • Time: O(m + n)
  • Space: O(1)
class Solution {
public:
    ListNode *getIntersectionNode(ListNode *headA, ListNode *headB) {
        ListNode* a = headA;
        ListNode* b = headB;

        while (a != b) {
            a = (a == nullptr) ? headB : a->next;
            b = (b == nullptr) ? headA : b->next;
        }
        return a; // either intersection node or nullptr
    }
};

Minimum Index Sum of Two Lists

Intitution:

  • We are given two lists of strings (restaurants)
  • Need to find common restaurants with minimum index sum (i + j)
  • Same restaurant can appear at different indices in both lists
  • Brute force would compare every pair → too slow
  • Key idea: quickly find index of a restaurant from one list while iterating the other

Key observation:

  • If we know index of a string in list1, we can compute sum instantly while scanning list2
  • Hashmap helps store string → index mapping for O(1) lookup
  • Track minimum index sum while iterating
  • Multiple answers possible if same minimum sum

Approach:

  • Build hashmap from list1 → (string → index)
  • Traverse list2
  • For each string:
    • If not in hashmap → skip
    • Else compute sum = index_in_list1 + current_index
    • If sum < min_sum:
      • clear result
      • add current string
      • update min_sum
    • Else if sum == min_sum:
      • add current string
  • Return result

Invariant:

  • At any point, result contains all strings with minimum index sum seen so far
  • min_sum always stores smallest (i + j) encountered
  • Hashmap ensures constant time lookup for matching strings

Edge cases (if any):

  • No common strings → return empty list
  • Multiple strings with same min sum → return all
  • One list empty → return empty
  • Same string at index 0 in both lists → immediate answer

Time & Space complexity:

  • Time: O(n + m)
  • Space: O(n)

Code

class Solution {
public:
    vector<string> findRestaurant(vector<string>& list1,
                                  vector<string>& list2) {
        vector<string> result;
        unordered_map<string, int> mp;

        for (int i = 0; i < list1.size(); ++i) {
            mp[list1[i]] = i;
        }

        int min_sum = INT_MAX;

        for (int i = 0; i < list2.size(); ++i) {
            auto it = mp.find(list2[i]);
            if (it == mp.end()) continue;

            int sum = it->second + i;

            if (sum < min_sum) {
                result.clear();
                result.push_back(list2[i]);
                min_sum = sum;
            } 
            else if (sum == min_sum) {
                result.push_back(list2[i]);
            }
        }

        return result;
    }
};

Addition of Numbers

Plus One Linked List

Code

class Solution {
    Node* reverseList(Node* head) {
        Node* prev = nullptr;
        Node* cur = head;
        while (cur != nullptr) {
            Node* nxt = cur->next;
            cur->next = prev;
            prev = cur;
            cur = nxt;
        }
        return prev;
    }

public:
    Node* addOne(Node* head) {
        head = reverseList(head);
        Node* cur = head;
        Node* prev = nullptr;
        int carry = 1;

        while (cur != nullptr && carry) {
            int sum = cur->data + carry;
            cur->data = sum % 10;
            carry = sum / 10;
            prev = cur;
            cur = cur->next;
        }

        if (carry) {
            prev->next = new Node(carry);
        }
        return reverseList(head);
    }
};

class Solution {

        ListNode dummy(0);
        ListNode* tail = &dummy;
        int carry = 0;
        while (l1 || l2 || carry) {
            int sum = carry;
            if (l1) {
                sum += l1->val;
                l1 = l1->next;

            if (l2) {
                sum += l2->val;
                l2 = l2->next;

            tail = tail->next;
        }
        return dummy.next;
    }

Reordering and Partitioning

Rotate List

Intitution:

  • Rotating right by k means last k nodes come to front
  • Instead of shifting nodes one by one, think in terms of cutting and reconnecting
  • Key trick: convert list into a circular list, then break at correct position

Key observation:

  • After rotation, new head = (len - k)th node
  • New tail = (len - k - 1)th node
  • If we connect tail → head, list becomes circular
  • Then just move to new tail and break

Approach:

  • Compute length of list and get tail
  • Do k = k % len (important for large k)
  • If k == 0 → return head
  • Make list circular → tail->next = head
  • Find new tail:
    • Move (len - k - 1) steps from head
  • New head = new_tail->next
  • Break circle → new_tail->next = NULL
  • Return new_head

Invariant:

  • After making circular list, every node is reachable in cycle
  • Moving (len - k - 1) steps always lands exactly at new tail
  • Breaking link restores proper singly linked list

Edge cases (if any):

  • head == NULL → return head
  • single node → no change
  • k % len == 0 → no rotation needed
  • k > len → must normalize using modulo

Time & Space complexity:

  • Time: O(n)
  • Space: O(1)
class Solution {
public:
    ListNode* rotateRight(ListNode* head, int k) {
        if (head == nullptr || head->next == nullptr) return head;
        int len = 1;
        ListNode* tail = head;
        while (tail->next != nullptr) {
            tail = tail->next;
            len++;
        }
        k %= len;
        if (k == 0) return head;
        tail->next = head;  // make circular
        int steps = len - k - 1;
        ListNode* new_tail = head;
        while (steps--) {
            new_tail = new_tail->next;
        }
        ListNode* new_head = new_tail->next;
        new_tail->next = nullptr;
        return new_head;
    }
};

Partition List

Intitution:

  • We need to rearrange nodes such that all nodes < x come before nodes >= x
  • Important: relative order inside each group must be preserved
  • This is not sorting, it is stable partitioning
  • Idea: build two separate lists and then join them

Key observation:

  • We cannot reorder nodes arbitrarily because order must be preserved
  • So instead of moving nodes around in-place, we collect:
    • nodes < x → left list
    • nodes >= x → right list
  • Finally connect left list → right list

Approach:

  • Create two dummy nodes: leftDummy and rightDummy
  • Maintain two tail pointers: leftTail and rightTail
  • Traverse original list:
    • store next pointer (important before breaking links)
    • detach current node (node->next = NULL)
    • if node->val < x → attach to leftTail and move tail
    • else → attach to rightTail and move tail
  • After traversal:
    • connect leftTail->next = rightDummy.next
  • Return leftDummy.next

Invariant:

  • left list always contains nodes < x in original order
  • right list always contains nodes >= x in original order
  • tails always point to last valid node in respective lists
  • no cycles because we break next pointers during processing

Edge cases (if any):

  • head == NULL → return NULL
  • all nodes < x → right list empty
  • all nodes >= x → left list empty
  • single node → unchanged

Time & Space complexity:

  • Time: O(n)
  • Space: O(1)

Code

class Solution {
public:
    ListNode* partition(ListNode* head, int x) {
        ListNode leftDummy(0), rightDummy(0);
        ListNode* leftTail = &leftDummy;
        ListNode* rightTail = &rightDummy;

        while (head != nullptr) {
            ListNode* nxt = head->next;
            head->next = nullptr;

            if (head->val < x) {
                leftTail->next = head;
                leftTail = leftTail->next;
            } else {
                rightTail->next = head;
                rightTail = rightTail->next;
            }

            head = nxt;
        }

        leftTail->next = rightDummy.next;
        return leftDummy.next;
    }
};

Swap nodes in pairs

Intitution:
We need to swap nodes in pairs in a linked list without modifying values.
Instead of changing values, we rewire pointers.
The key is to operate locally on 2 nodes at a time and maintain connection with the rest of the list.

Key observation:
Each swap only affects 3 pointers:
prev -> first -> second -> nextPair
After swap:
prev -> second -> first -> nextPair

We only need to update:

  1. first->next
  2. second->next
  3. prev->next

Approach:

  • Use a dummy node before head to simplify edge handling
  • Maintain a pointer prev which always points to node before current pair
  • For each iteration:
    • Identify:
      • first = prev->next
      • second = first->next
    • Perform pointer rewiring:
      • first->next = second->next
      • second->next = first
      • prev->next = second
    • Move prev to first (new tail of swapped pair)
  • Continue while at least 2 nodes exist

Invariant:

  • prev always points to the node just before the pair being processed
  • The part before prev is always correctly swapped
  • Remaining list starting at prev->next is untouched

Edge cases (if any):

  • Empty list → return head
  • Single node → no swap needed
  • Odd number of nodes → last node remains as it is

Time & Space complexity:

  • Time: O(n) (each node visited once)
  • Space: O(1) (in-place pointer manipulation)
class Solution {
public:
    ListNode* swapPairs(ListNode* head) {
        ListNode dummy(0, head);
        ListNode* prev = &dummy;

        while (prev->next && prev->next->next) {
            ListNode* first = prev->next;
            ListNode* second = first->next;

            first->next = second->next;
            second->next = first;
            prev->next = second;

            prev = first;
        }
        return dummy.next;
    }
};

Reorder List

Intitution:
This problem looks hard only because the required final order is unusual:
L0 -> Ln -> L1 -> Ln-1 -> L2 -> Ln-2 ...
The important idea is to not try to build this order directly from the original list in one step.
Instead, break the problem into 3 standard linked list operations:

  • find the middle
  • reverse the second half
  • merge the two halves alternately

Why this works:

  • The first half already gives nodes in correct left-to-right order: L0, L1, L2 ...
  • The second half, if reversed, gives nodes in required right-to-left order: Ln, Ln-1, Ln-2 ...
  • After that, we just weave the two lists together:
    first node from first half, first node from reversed second half, next node from first half, next node from reversed second half, and so on

Example:

  • Original: 1 -> 2 -> 3 -> 4 -> 5
  • First half: 1 -> 2 -> 3
  • Second half: 4 -> 5
  • Reverse second half: 5 -> 4
  • Merge alternately:
    1 -> 5 -> 2 -> 4 -> 3

So the real trick is:

  • convert the list into two useful forms
  • then interleave them

Key observation

  • Direct simulation is messy
  • Repeatedly finding the last node would take O(n^2)
  • If we reverse the second half once, the reorder becomes a simple alternating merge
  • We must disconnect the first half from the second half before reversing, otherwise old links remain and can create cycles
  • The best interview solution is:
    • slow/fast pointer to find middle
    • iterative reverse
    • iterative merge
  • This gives O(n) time and O(1) extra space

Approach

  1. Find the end of first half
  • Use slow and fast pointers
  • Move slow by 1 step and fast by 2 steps
  • When loop ends, slow will be at the end of first half
  1. Split the list
  • Let second = slow->next
  • Set slow->next = nullptr
  • Now we have two independent lists
  1. Reverse the second half
  • Reverse the list starting from second
  • After reversal, nodes appear in required tail-first order
  1. Merge alternately
  • Use two pointers: first and second
  • Save next pointers before changing links
  • Attach one node from first half, then one from second half
  • Continue until second half is exhausted

Invariant

  • Before reversing:
    • first half is intact
    • second half starts at slow->next
  • After splitting:
    • first half and second half are independent lists
  • During reverse:
    • prev always points to reversed part
    • cur always points to remaining unreversed part
  • During merge:
    • all nodes before first are already reordered correctly
    • first and second point to the next nodes to weave
  • At every step, no nodes are lost because next pointers are saved first

Edge cases (if any)

  • Empty list
    • nothing to reorder
  • Single node
    • already reordered
  • Two nodes
    • remains same after reorder
  • Odd length list
    • first half will have one extra node
    • final middle node stays at the end
  • Even length list
    • both halves merge evenly
  • Important bug:
    • if you do not cut with slow->next = nullptr, merge can create cycles

Time & Space complexity

  • Find middle: O(n)
  • Reverse second half: O(n)
  • Merge halves: O(n)
  • Total time: O(n)
  • Extra space: O(1)

Code

class Solution {
    ListNode* reverseList(ListNode* head)
    {
        ListNode* prev = nullptr;
        ListNode* cur = head;

        while(cur != nullptr)
        {
            ListNode* nxt = cur->next;
            cur->next = prev;
            prev = cur;
            cur = nxt;
        }
        return prev;
    }

public:
    void reorderList(ListNode* head) {
        if(head == nullptr || head->next == nullptr) return;

        // 1. Find end of first half
        ListNode* slow = head;
        ListNode* fast = head;

        while(fast->next != nullptr && fast->next->next != nullptr)
        {
            slow = slow->next;
            fast = fast->next->next;
        }

        // 2. Split and reverse second half
        ListNode* second = slow->next;
        slow->next = nullptr;
        second = reverseList(second);

        // 3. Merge alternately
        ListNode* first = head;
        while(second != nullptr)
        {
            ListNode* nxt1 = first->next;
            ListNode* nxt2 = second->next;

            first->next = second;
            second->next = nxt1;

            first = nxt1;
            second = nxt2;
        }
    }
};
