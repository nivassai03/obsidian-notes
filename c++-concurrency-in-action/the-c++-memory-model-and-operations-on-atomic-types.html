The C++ Memory Model And Operations On Atomic Types

Memory Model and Atomic Operations (Chapter 5 Notes)

Executive Summary: C++11 introduced a formal memory model defining how threads interact via shared memory. Chapter 5 covers this model and atomic operations. Key points: every variable is an object occupying one or more memory locations; unsynchronized concurrent access to the same location with a write is a data race (undefined behavior). Using std::atomic types or mutexes provides synchronization. Atomic operations can be ordered with memory-order tags (seq_cst, acquire, release, etc). A happens-before relation (combining sequenced-before and inter-thread synchronizes-with) determines visibility and ordering across threads. Sequential consistency (SC) is the strongest model (a single total order); weaker models (TSO, PSO, ARM) allow various reorderings. Acquire-release synchronizes pairs of operations (e.g. store-release with load-acquire) to propagate writes. Fences (atomic_thread_fence) act as memory barriers, enforcing ordering on relaxed atomics. The chapter gives examples (spinlock, queue counter), litmus tests, and best practices (avoid data races, prefer acquire/release for performance, use SC for simplicity) with formal definitions and diagrams.

5.1 Memory Model Basics

  • Objects and memory locations: C++ defines an object as “a region of storage”. Every variable (including members of other objects) is an object. A memory location is defined (for concurrency) as either an entire scalar object or a sequence of adjacent bit-fields. Each fundamental object (e.g. int, char) occupies exactly one memory location. Adjacent bit-fields share a location. These rules imply, for example, that two distinct int variables are in separate locations (no race), but two atomic fields in the same int might share a location (potential race).
  • Concurrent accesses: If two threads access different memory locations, their operations never conflict (no ordering needed). If they access the same location, synchronization is required. If at least one access is a write and there is no “happens-before” ordering between them, this is a data race, causing undefined behavior. Put differently, any pair of concurrent conflicting accesses must be ordered (by locks or atomics) to avoid UB. For example, without atomic or mutex coordination, two threads concurrently writing and reading the same int leads to a data race.
  • Avoiding races: Introducing a synchronization (e.g. a mutex lock/unlock) orders the accesses. Alternatively, using atomic operations on the same location avoids undefined behavior: even if a “race” still occurs (which atomic happens first is indeterminate), it is defined as a well-formed atomic interleaving. In other words, non-atomic writes cause UB, but atomic writes do not (they may still contend but result in one taking effect in a defined modification order).
  • Modification order: Each atomic object has a modification order: a total order of all its write (store/RMW) operations in the execution, starting from its initialization. All threads must agree on this order (i.e. it’s coherent). For non-atomic objects, the programmer must ensure through synchronization that threads see a single consistent modification order. If different threads perceive different sequences of values for a variable, that indicates a data race. The model forbids reordering that would violate this: once a thread reads a particular point in the modification order, later reads must see that value or a subsequent one.

5.2 Atomic Operations and Types in C++

  • Atomic types in <atomic>: C++ provides special atomic types (in <atomic>). All operations on these types are atomic (as per the language definition). Key types include:
    • std::atomic_flag: The simplest atomic boolean (two states: set or clear). It must be initialized with ATOMIC_FLAG_INIT (always starts clear). It is guaranteed lock-free and supports only test_and_set() (set flag and return old value) and clear() (clear flag) (optionally with memory-order arguments).
    • std::atomic<bool>: A full-featured atomic boolean. Can be constructed or assigned from a normal bool. Supports store(), load(), exchange(), compare_exchange_weak/strong(), etc., returning plain bool values (assignment returns the stored value). For example, b = false; is equivalent to b.store(false, mem_order_seq_cst).
    • std::atomic<T*>: An atomic pointer type. Supports all the above (load/store/exchange/compare) returning T*. Additionally, supports pointer arithmetic: fetch_add(n), fetch_sub(n), and the operators +=, -=, ++, --. E.g. p.fetch_add(3) adds 3 elements to p and returns the old pointer.
    • Atomic integral types (std::atomic<int>, etc): In addition to load/store/exchange/compare, they support arithmetic atomics: fetch_add/sub/and/or/xor, the compound assignments (+=, -=, &=, |=, ^=), and pre- and post-increment/decrement. The named fetch_* functions return the old value; operators like x++ return the new value. (Missing are multiply/divide/shift, but those can be implemented with CAS loops if needed.)
    • Primary template std::atomic<UDT>: Allows atomic variants of user-defined types (UDT). The UDT must be trivially copyable (trivial copy-assignment, no virtuals). The interface is the same as atomic<bool> (load/store/exchange/compare, etc), but with UDT values. The compiler will typically use memcpy or locks internally as needed.
  • Member and free functions: Each atomic type has member functions (e.g. x.load(), x.store(v), etc) and there are equivalent non-member functions like std::atomic_load(&x), std::atomic_store(&x,v), std::atomic_exchange(&x,v), etc.. These free functions take a pointer to the atomic object and (optionally) a memory-order tag. E.g. std::atomic_load_explicit(&x, mem_order_acquire) does the same as x.load(mem_order_acquire). Table 5.3 in the book (p. 164) enumerates which operations each atomic type supports.

5.3 Synchronizing Operations and Enforcing Ordering

5.3.1 The synchronizes-with relationship

Certain atomic operations create a synchronizes-with edge between threads. The core rule: if thread A does an atomic store with release semantics on X, and thread B does an atomic load with acquire semantics on X that reads the value written by A, then the store synchronizes-with that load. (Release-store and Acquire-load establish this; seq_cst also counts as both release and acquire.) In effect, a store(..., memory_order_release) in thread A makes its prior writes visible to thread B once a load(..., memory_order_acquire) in B reads that store’s value. More generally, C++ allows a store-release to synchronize with a load-acquire even if intervening operations (like RMWs) occur on the same atomic (the release sequence concept). Mutex unlock (release) and lock (acquire) also imply synchronizes-with: the unlock operation synchronizes-with the subsequent lock by another thread.

5.3.2 The happens-before relationship

The happens-before relation is the transitive closure of program order (“sequenced-before” within a thread) and inter-thread synchronizes-with. Concretely:

  • Program order: Within one thread, if statement A is before B in the source code (or more formally sequenced before), then A happens-before B. Thus all operations in one statement happen-before any in the next.
  • Inter-thread: If A synchronizes-with B, then A inter-thread-happens-before B. This is transitive: if A hb B and B hb C, then A hb C.
  • Combined: happens-before allows chaining between threads. For example, if modifications_to_data(); unlock(); in thread 1 and lock(); use_data(); in thread 2, then the stores before unlock() happen-before the stores/loads after the matching lock() (via unlock→lock synchronizes-with).
  • Strongly-happens-before: If consume-acquire were used, there is a weaker “dependency-ordered-before” that only applies along data dependencies. However, C++17 discourages memory_order_consume, so effectively C++ uses acquire for practical purposes. We will use “happens-before” to mean either sequenced-before or an inter-thread chain. Crucially, happens-before is guaranteed acyclic; this consistency requirement forbids cycles like “A hb B and B hb A”.

5.3.3 Memory ordering for atomic operations

C++11 defines six memory orders for atomics: relaxed, consume (deprecated), acquire, release, acq_rel, and seq_cst. These map into three conceptual models:

  • Sequentially consistent (SC) ordering (memory_order_seq_cst): The default. All SC operations on atomics appear to execute in a single global sequence that is consistent with the program order of each thread. This means if all threads use SC atomics, the execution is like interleaving instructions from a single thread (no reordering across threads). For example, in Listing 5.4 below (using seq_cst for all ops) one can show that the final assertion must succeed (at least one load sees a 1). In SC mode, a store-seq_cst synchronizes with a load-seq_cst that reads that value, and later SC accesses cannot be reordered before earlier ones. The benefit is simplicity: easy to reason about. The cost is performance: on weak hardware, enforcing a global order may require expensive fences or cache syncs. (x86/TSO, however, largely provides SC cheaply for loads, though stores need an mfence for full seq_cst.)

  • Acquire-release ordering: This weaker model has no global total order. A store(..., memory_order_release) synchronizes-with a matching load(..., memory_order_acquire) (as above). Acquire-release ensures that within each thread, operations before a release store are not moved after it, and operations after an acquire load are not moved before it. However, different threads may see different orders for unrelated atomics. Example: Listing 5.7 (below) uses release stores and acquire loads on x and y. Even though each pair of store/release and load/acquire forms a sync, there is no total global order, so it is possible for both reader threads to see stale values (z could remain 0). In contrast, SC would forbid that outcome. Acquire-release is faster (no global fence needed) but requires care: you must explicitly insert release on writers and acquire on readers to get the needed guarantees.

  • Relaxed ordering (memory_order_relaxed): Only guarantees atomicity (each operation is indivisible) but no ordering guarantees between threads. Relaxed loads/stores on different variables can be freely reordered by compiler or hardware. Use relaxed only when you need atomic reads/writes but no synchronization (e.g. atomic counters where order doesn’t matter).

  • Memory fences: C++ provides atomic_thread_fence(order) for additional barriers. A fence enforces ordering on previous/following operations in that thread. For example, a atomic_thread_fence(memory_order_release) makes all prior stores visible before the fence, and an ..._acquire fence makes all following loads see after the fence. A release fence in one thread synchronizes with an acquire fence in another (if they use the same atomic variable). In practice, fences let you order non-atomic or relaxed accesses: see section 5.3.6 below.

5.3.4 Release Sequences and Extended Synchronizes-With

A release sequence is a chain of atomic operations on the same object: it starts with a release-store (or an atomic RMW with release) and continues with zero or more read-modify-write ops (by any thread), each reading the value from the previous step. The key rule: the first store in the sequence synchronizes-with an acquire load that reads a value from any operation in that sequence. Concretely, in the queue example (Listing 5.11) the producer does count.store(N, release), and each consumer does count.fetch_sub(1, acquire). The first fetch_sub also has release semantics (because fetch-subtract is an RMW with memory_order_acq_rel by default), forming a release sequence. Thus the initial store(N,release) synchronizes with a later consumer’s fetch_sub(acquire) even though another RMW intervened. This ensures that when a consumer sees a positive count, it also sees all prior queue writes. (Without this chain, a second consumer might not see the producer’s store if it only saw the first consumer’s write.)

5.3.5 Fences (Memory Barriers)

Fences are special atomic operations that impose ordering constraints on other operations in the same thread, without touching a memory value. A fence acts as a barrier: no read/write can move past it in forbidden ways. Common usage: pair a release fence in one thread with an acquire fence in another. For example, consider two threads writing x then y with a release fence between them, while two readers load y then x with an acquire fence between. The release fence on thread A synchronizes with the acquire fence on thread B (assuming the loads read the same atomic), establishing happens-before between x.store and x.load. In short, atomic_thread_fence(mem_order_release) ensures all prior memory ops are visible before the fence, and ..._acquire ensures subsequent ops see effects after the fence. A seq_cst fence is a full barrier in both directions. Fences let you enforce ordering for relaxed atomics or non-atomics when needed.

5.3.6 Ordering Non-atomic Operations via Atomics

Non-atomic accesses have no built-in synchronization. However, if non-atomic operations are sequenced-before a synchronization operation (lock/unlock or an atomic fence) in one thread, and another thread sees that synchronization, then the non-atomic ops become ordered. Example: In the spinlock below, non-atomic data writes are done before unlock(), and unlock() (release) synchronizes-with the other thread’s lock() (acquire), so the data writes happen-before the second thread’s data reads. Likewise, even ordinary bools can be ordered by atomics: if a thread does x = true; atomic_thread_fence(release); and another does atomic_thread_fence(acquire); if (x) ..., then the write to x is guaranteed visible after the acquire fence (just as if x were atomic). In general, use atomic ops or fences whenever you need to order any shared memory operations.

5.3.7 Mutexes and High-level Sync

All standard synchronization primitives (mutexes, condition variables, etc.) build on these atomic rules. For instance, std::mutex::lock is defined to acquire the mutex, and unlock to release it, yielding the same happens-before guarantees. As one result, any data protected by a mutex behaves as if there is a release-store on unlock and an acquire-load on subsequent lock. The chapter notes that the C++ standard library documentation specifies many such synchronizes-with guarantees (e.g. thread.join(), shared_mutex lock/unlock, futures, etc.), all ultimately relying on atomics with memory-order semantics.

Formal Definitions and Properties

  • Data race ⇒ Undefined Behavior: In C++’s model, the presence of a data race makes the entire execution undefined. This is a form of UB as specified by the standard. Thus any reasoning about concurrent code must eliminate data races.
  • Modification order (atomic coherence): For each atomic object M, C++ guarantees a total modification order of all writes to M. The following coherence rules hold for atomic operations (from cppreference):
    1. Write–write coherence: If write A to M happens-before write B to M, then A precedes B in M’s modification order.
    2. Write–read coherence: If write A to M happens-before a read B of M, then B sees A or a later write.
    3. Read–write coherence: If read A of M happens-before write B to M, then A sees a write X that precedes B in the modification order.
    4. Read–read coherence: If read A of M happens-before read B of M, and A saw write X, then B sees either X or a later write.
      These ensure that all threads have a coherent view of each atomic variable.
  • Happens-before is transitive and acyclic: If A happens-before B and B happens-before C, then A happens-before C. This acyclicity (no circular causality) is required by the model (the compiler may insert fences to enforce it if necessary).
  • Release sequence (formal): A release sequence is, formally, a chain of one release or RMW followed by dependent RMWs. It ensures that if a load (acquire) reads from anywhere in the sequence, the head synchronizes with that load.
  • Sequential Consistency theorem: Under SC ordering, the result of a multi-threaded program is as if all atomic operations were executed in some single sequence consistent with each thread’s program order. (Thus one can enumerate all interleavings and eliminate those that contradict the per-thread order.)
  • Memory order guarantees:
    • memory_order_seq_cst imposes one global order on all seq_cst atomics.
    • memory_order_acquire: a load-acquire ensures no later memory accesses are reordered before it, and sees all writes that happened-before the matching release.
    • memory_order_release: a store-release ensures no prior memory accesses are reordered after it, and makes all previous writes visible before it to other threads.
    • memory_order_relaxed: no cross-thread ordering (only atomicity).
    • Fences behave like global versions of these constraints.
  • C++ vs hardware models: The C++ memory model abstracts the language guarantees; actual hardware (TSO, ARM, POWER) may allow more reordering. C++ compilers map atomic orders to hardware fences as needed. For example, x86’s TSO allows a store-load reorder (writes buffered), so C++ compilers insert an mfence for seq_cst or use LOCK-prefixed instructions for RMW to ensure semantics. On ARM/POWER, additional barriers are needed for acquire/release.

Code Examples

  • Spinlock with acquire-release: An example mutex using atomic_flag: 

    struct spinlock_mutex {
    std::atomic_flag flag = ATOMIC_FLAG_INIT;
    void lock() {
        while(flag.test_and_set(std::memory_order_acquire))
            ; // spin until flag was clear
    }
    void unlock() {
        flag.clear(std::memory_order_release);
    }
};

    Here, lock() uses test_and_set(acquire) and unlock() uses clear(release). The first thread to lock() will clear the flag and exit; others will spin. Crucially, unlock() (release) synchronizes-with the next thread’s lock() (acquire). Thus any writes done before unlock() are visible after lock() in the other thread. For example, data writes before unlock() (sequenced-before) happen-before data reads after the second thread’s lock() (because unlock() release → lock() acquire synchronizes). This ensures safe access to protected data.

  • Sequentially-consistent litmus (Listing 5.4): Consider two threads setting flags x and y, and two readers: 

  std::atomic<bool> x, y;
  std::atomic<int> z;
  void write_x() { x.store(true, std::memory_order_seq_cst); }   // ①
  void write_y() { y.store(true, std::memory_order_seq_cst); }   // ②
  void read_x_then_y() {
      while(!x.load(std::memory_order_seq_cst));                // ③
      if(y.load(std::memory_order_seq_cst)) ++z;               // ④
  }
  void read_y_then_x() {
      while(!y.load(std::memory_order_seq_cst));                // ⑤
      if(x.load(std::memory_order_seq_cst)) ++z;               // ⑥
  }
  int main() {
      x = false; y = false; z = 0;
      std::thread a(write_x), b(write_y), c(read_x_then_y), d(read_y_then_x);
      a.join(); b.join(); c.join(); d.join();
      assert(z.load() != 0);  // Under seq_cst, this assert **cannot** fire.
  }

Explanation: Because we use SC stores and loads, all atomic operations form one total order. Either x.store happens before y.store or vice versa in that order. In either case, one reader will see the other thread’s write and increment z. The book confirms “the assert can never fire” under SC.

  • Acquire-release litmus (Listing 5.7): Change the stores/loads to use release/acquire: 
  // (Similar setup)
  void write_x() { x.store(true, std::memory_order_release); }
  void write_y() { y.store(true, std::memory_order_release); }
  void read_x_then_y() {
      while(!x.load(std::memory_order_acquire));
      if(y.load(std::memory_order_acquire)) ++z;
  }
  void read_y_then_x() {
      while(!y.load(std::memory_order_acquire));
      if(x.load(std::memory_order_acquire)) ++z;
  }
  // ... same main ...
      assert(z.load() != 0);  // Now this **can** fail (allowed by acquire-release).

Explanation: With only acquire/release, there is no global order between the two stores. It is possible both reader threads see only the initial false values (so z remains 0). Indeed, the text notes that in this case “the assert can fire… possible both loads read false”. This litmus illustrates that acquire-release alone does not enforce sequential consistency.

Comparative Tables

Atomic types & operations:

Type (T) Operations
std::atomic_flag test_and_set([mem_order]) (atomic set+test), clear([mem_order]). Always lock-free.
std::atomic<bool> load, store, exchange, compare_exchange_weak/strong, and (implicitly) assignment from bool. No arithmetic.
std::atomic<T*> Same as bool plus pointer arithmetic: fetch_add/sub(n), +=, -=, ++, --.
std::atomic<Integral> Same as bool plus integer operations: fetch_add, fetch_sub, fetch_and/or/xor, compound assigns (+=, etc), ++, --.
std::atomic<UDT> (triv.) If UDT is trivially copyable, supports load/store/exchange/compare (read-modify if defined). Acts like an atomic sequence of bytes.

Memory orders:

memory_order Guarantees
seq_cst (default) Strongest: one total order of all seq_cst ops; each store-release→load-acquire sync is also a seq_cst sync. No reordering across threads.
acquire Load-acquire: no following accesses move before it; sees all prior stores (in other thread) released on same atomic.
release Store-release: no preceding accesses move after it; makes all prior stores visible (to other threads loading this atomic).
acq_rel Both acquire & release (for RMW ops).
consume (deprecated) Load-consume: like acquire but only enforces order on dependent operations. Rarely used.
relaxed Only atomicity (no ordering): operations on different vars can reorder freely.
Fence (atomic_thread_fence(order)) Memory barrier: release-fence orders prior stores before it; acquire-fence orders later loads after it; seq_cst fence does both. A release fence synchronizes with a later acquire fence.

Consistency models:

Model Behavior Example (hardware)
Sequential Consistency (SC) All operations appear in a single interleaving consistent with program order. All threads see same order. (Theoretical ideal; x86 with locks/mfence can approximate)
Total Store Order (TSO) Writes may be buffered, so a thread might see its own write immediately, but others see it later. SC except store→load reorder allowed. Intel x86, SPARCv8 TSO
Partial Store Order (PSO) Like TSO but allows a write to be seen by some cores before others (write non-atomic), and allows more reorder (like ARM). (Some SPARC or Power variants)
Weak (Relaxed) models Loads and stores may be reordered or speculated arbitrarily; writes may become visible to different threads at different times. Requires explicit fences to enforce order. ARM, POWER, modern GPUs.

Common Pitfalls & Best Practices

  • Never rely on non-atomic shared writes: Without atomic or lock, writes can be reordered or lost; any concurrent write+read is a data race (UB). Always protect shared data with atomics or mutexes.
  • Mind evaluation order: Even within one thread, function argument evaluation (or multiple writes in one statement) is unspecified. E.g. foo(get_num(), get_num()) may print in any order (as shown in Listing 5.3). Use sequencing or intermediate variables if order matters.
  • Choose appropriate memory order: Use seq_cst if unsure (easiest reasoning). For performance, use acquire/release where possible and verify correctness. Avoid relaxed unless ordering truly doesn’t matter. (E.g. use relaxed only for counters with no other communication.)
  • Avoid memory_order_consume: It’s deprecated and not reliably implemented (usually treated as acquire). Use acquire.
  • Prevent compiler reordering: If needed, use atomic_thread_fence or volatile (only as I/O barrier, NOT as thread sync). In C++, volatile does not impose memory order between threads.
  • Compile-time checks: Atomic operations must not be copied (e.g. atomic_flag has no copy constructor). User-defined std::atomic<T> requires trivial copy semantics.
  • Lock-free checks: Many atomics have .is_lock_free(). If an atomic is not lock-free on a platform, the compiler uses a mutex internally (performance penalty).
  • Avoid out-of-thin-air values: The model forbids speculative creation of arbitrary values. Properly synchronized code (via happens-before) ensures that loads see only “real” stored values.

For quick reference, recall:

  • Release store + Acquire load = synchronization (→happens-before).
  • Seq_cst operations obey a single order.
  • Relaxed = “no guarantees” (only atomicity).
  • Data race = undefined behavior – avoid at all costs.
