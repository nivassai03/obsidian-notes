Synchronising Concurrent Operations

Waiting for an event or other condition

  • C++ standard provided futures and conditional variables for synchronisation b/w threads instead of just protecting the data.
  • Conditional variable is associated with a event or a other condition, one or more threads can wait for that event to be finished. It will notify one or more threads if that condition is satisfied.
std::mutex mut;
std::queue<data_chunk> data_queue;                 
std::condition_variable data_cond;
void data_preparation_thread()
{
    while(more_data_to_prepare())
    {
        data_chunk const data=prepare_data();
        {
            std::lock_guard<std::mutex> lk(mut);
            data_queue.push(data);                 
        }
        data_cond.notify_one();                    
    }
}
void data_processing_thread()
{
    while(true)
    {
        std::unique_lock<std::mutex> lk(mut);      
        data_cond.wait(
            lk,[]{return !data_queue.empty();});   
        data_chunk data=data_queue.front();
        data_queue.pop();
        lk.unlock();                               
        process(data);
        if(is_last_chunk(data))
            break;
    }
}
  • C++ std provided std::conditional_variable which works with only std::mutex and std::conditional_variable_any which works with any mutex like.
  std::conditional_variable(lock_object,lambda (predicate));
  • The implementation of wait() returns if the predicate is satisfied.
  • If the condition is not satisfied wait() unlocks the mutex and put the thread in a blocked or waiting state.
  • When a conditional variable is notified by a call to notify_one() by another thread the thread wakes from slumber reacquires the lock on the mutex and checks the condition again returning from wait with mutex locked if condition is satisfied.
  • During a call to wait() a cv can check the supplied condition any number of times but it does by locking the mutex.
  • A spurious wake happens when a waiting thread reacquires the mutex and checks the condition with out notification from another thread.
  • cv::wait() is optimisation over a busy wait.

Thread safe queue with conditional variable

#include <queue>
#include <memory>
#include <mutex>
#include <condition_variable>
template<typename T>
class threadsafe_queue
{
private:
    mutable std::mutex mut;           
    std::queue<T> data_queue;
    std::condition_variable data_cond;
public:
    threadsafe_queue()
    {}
    threadsafe_queue(threadsafe_queue const& other)
    {
        std::lock_guard<std::mutex> lk(other.mut);
        data_queue=other.data_queue;
    }
    void push(T new_value)
    {
        std::lock_guard<std::mutex> lk(mut);
        data_queue.push(new_value);
        data_cond.notify_one();
    }
    void wait_and_pop(T& value)
    {
        std::unique_lock<std::mutex> lk(mut);
        data_cond.wait(lk,[this]{return !data_queue.empty();});
        value=data_queue.front();
        data_queue.pop();
    }
    std::shared_ptr<T> wait_and_pop()
    {
        std::unique_lock<std::mutex> lk(mut);
        data_cond.wait(lk,[this]{return !data_queue.empty();});
        std::shared_ptr<T> res(std::make_shared<T>(data_queue.front()));
        data_queue.pop();
        return res;
    }
    bool try_pop(T& value)
    {
        std::lock_guard<std::mutex> lk(mut);
        if(data_queue.empty())
            return false;
        value=data_queue.front();
        data_queue.pop();
        return true;
    }
    std::shared_ptr<T> try_pop()
    {
        std::lock_guard<std::mutex> lk(mut);
        if(data_queue.empty())
            return std::shared_ptr<T>();
        std::shared_ptr<T> res(std::make_shared<T>(data_queue.front()));
        data_queue.pop();
        return res;
    }
    bool empty() const
    {
        std::lock_guard<std::mutex> lk(mut);
        return data_queue.empty();
    }
};
  • notify_all() causes all the threads to currently executing wait to check the condition they are waiting for.
  • A condition variable might not be best choice for synchronisation if the waiting thread is going to wait only once. This is especially true if the condition being waited on is the availability of data.

Waiting for one of events with futures

  • C++ standard models this sort of one-off event with something called future.
  • C++ has two sort of futures. std::future (unique future) and std::shared_future (shared future).
  • std::future is one and only instance that refers to its associated event.
  • Multiple instances of std::shared_future may refer to the same event. All instances will become ready at the same time and may access any data associated with the event.
  • std::future<void> and std::shared_future<void> should be used when there is no associated data.
  • Future objects are not thread safe must be protected with synchronisation mechanisms.
  • Multiple threads may access their own copy of std::shared_future with out synchronisation.

Returning values from background tasks

  • std::thread doesn't provide a direct mechanism for transferring the result back.
  • std::async can be used to start the asynchronous task for which we don't need the result right away.
  • std::async returns a std::future object which will eventually hold the return value of the function.
  • get() is called on future, and thread is blocked until the future is ready and then return the value.
  • std::async allows you to pass arguments to functions similar way that std::thread does.
  • By default its upto the implementation whether std::async starts a new thread or whether task runs synchronously when the future is waited for.
  • std::async takes additional parameters of the type std::launch before the function call.
  • std::launch::deferred | std::launch::async (default) to indicate the implementation may choose
  • std::launch::deferred to indicate that the function call is to be deffered until either wait() or get().
  • std::launch::async to indicate that the function must run on its own thread.

Associating a task with future

  • std::packaged_task ties a future with a callable object. When invoked it calls the associated function or callable object and make future ready.
  • Template parameter for packaged_task<> class template is function template.
  • The return type of function signature indentifies the type of the std::future returned from the get_future().
  • The argument list of function signature is used to specify the signature of the packaged_tasks function call operator.
  • Below shows an example program
#include <deque>
#include <mutex>
#include <future>
#include <thread>
#include <utility>
std::mutex m;
std::deque<std::packaged_task<void()> > tasks;
bool gui_shutdown_message_received();
void get_and_process_gui_message();
void gui_thread()                               
{
    while(!gui_shutdown_message_received())     
    {
        get_and_process_gui_message();          
        std::packaged_task<void()> task;
        {
            std::lock_guard<std::mutex> lk(m);
            if(tasks.empty())                   
                continue;
            task=std::move(tasks.front());      
            tasks.pop_front();
        }
        task();                                 
    }
}
std::thread gui_bg_thread(gui_thread);
template<typename Func>
std::future<void> post_task_for_gui_thread(Func f)
{
    std::packaged_task<void()> task(f);        
    std::future<void> res=task.get_future();    
    std::lock_guard<std::mutex> lk(m);
    tasks.push_back(std::move(task));          
    return res;                                
}

Making (std::)promises

  • std::promise<T> provides a means of setting a value that can be latter read through a associated std::future object.
  • You can obtain a std::future object associated with a given std::promise by calling the get_future() member function.
#include <future>
void process_connections(connection_set& connections)
{
    while(!done(connections))                           
    {
        for(connection_iterator                          
                connection=connections.begin(),end=connections.end();
            connection!=end;
            ++connection)
        {
            if(connection->has_incoming_data())         
            {
                data_packet data=connection->incoming();
                std::promise<payload_type>& p=
                    connection->get_promise(data.id);    
                p.set_value(data.payload);
            }
            if(connection->has_outgoing_data())          
            {
                outgoing_packet data=
                    connection->top_of_outgoing_queue();
                connection->send(data.payload);
                data.promise.set_value(true);            
            }
        }
    }
}

Saving an exception for future

  • If a function call invoked as a part of async throws an exception. The exception is stored in the future in place of stored value.
  • When the future becomes ready and a call to get() re throws the store exception.
  • Same thing happens with std::packaged_task() when the wrapped function throws an exception that exception is stored in the future in place of result.
  • Similarly std::promise provides some functionality with explicit function call set_exception()
  • Another way to store an exception in a future is to destroy the std::promise or std::packaged_task associated with the future is without calling either of the set functions on the promise or invoking the packaged task.
  • In the above case the destructor of the std::promise or std::packaged_task will store a std::future_error with an error code of std::future_error::broken_promise in the associated state if the future isn't already ready.
some_promise.set_exception(std::make_exception_ptr(std::logic_error("foo ")));

Waiting for multiple threads

  • Calls to the member functions of a particular std::future instance are not synchronised with each other.
  • std::future models unique ownership of the asynchronous result.
  • std::future is only movable. Ownership can be transferred but only one refers to a particular asynchronous result at a time.
  • std::shared_future instances are copyable.
  • std::shared_future member functions on an individual objects are still unsynchronised.
  • The preferred way to use it would be pass a copy of the std::shared_future object to each thread. Each thread can access its own local shared future object safely.
  • Accessing the shared asynchronous state from multiple threads is safe if each thread access that state through its own std::shared_future object.
  • std::future has a share() member function that creates a new std::shared_future and transfers the ownership to it directly.

Working with time limit

  • All blocking calls will wait for an indefinite period of time.
  • Timeouts are of two types.
    • Duration based (function call have _for suffix)
    • Absolute point (function call have _until suffix)
  • Conditional variable have two overload. wait_for() and wait_until()

Clocks

  • If a clock ticks at a uniform rate and can't be adjusted it is Steady Clock.
  • C++ standard library provides Steady Clock ,System Clock (real time) and High resolution clock.

Duration's

  • C++ standard provides std::chrono::duration<>. First template parameter is the type of representation (int,double,long etc). Second is fraction specifying how many seconds each unit of the duration represents.
    • std::chrono::duration<short,std::ratio<60,1>> represents minutes.
  • Conversion is implicit where it does not require truncation of the value.
    • Hours -> Seconds (OK)
    • Seconds -> hours (NOT OK)
  • Duration supports arithmetic.
  • The first template arguments in duration represents the maximum amount of time we can represent with that storage.
  • Wait functions returns a status to indicate whether wait timed out or whether wait for event occurred.
    • std::future_status::timeout
    • std::future_status::ready
    • std::future_status::deferred

Timepoints

  • The value of time point is the length of time since a specific point in time called the epoch of the clock.
  • C++ standard provides std::chrono::timepoint<>
    • The first template argument is which clock it refers to.
    • The second template argument is units of measurement (specialisation of std::chrono::duration)
    • Ex. std::chrono::timepoint<std::chrono::system_clock,std::chrono::minutes>
  • When you pass the timepoint to a wait function that takes an absolute timeout, the clock parameter of the timepoint is used to measure the time.

Using Synchronization Of Operations To Simplify Code

  • Instead of sharing data directly between threads each task can be provided with the data it needs and the result can be disseminated to any other thread.

  • A future can be passed around between threads to allow the result of one computation to depend on the result on the another without any explicit access to shared data.

  • CSP (Communicating Sequential Process) is another paradigm where the threads are conceptually entirely separate with no shared data but with communication channels that allows messages to be passed between them (erlang).

  • Sequential quick sort

template<typename T>
std::list<T> sequential_quick_sort(std::list<T> input)
{
    if(input.empty())
    {
        return input;
    }
    std::list<T> result;
    result.splice(result.begin(),input,input.begin());          
    T const& pivot=*result.begin();                              

    auto divide_point=std::partition(input.begin(),input.end(),
            [&](T const& t){return t<pivot;});                   
    std::list<T> lower_part;
    lower_part.splice(lower_part.end(),input,input.begin(),
        divide_point);                                           
    auto new_lower(
        sequential_quick_sort(std::move(lower_part)));           
    auto new_higher(
        sequential_quick_sort(std::move(input)));                
    result.splice(result.end(),new_higher);                      
    result.splice(result.begin(),new_lower);                     
    return result;
}
  • Parallel quick sort
template<typename T>
std::list<T> parallel_quick_sort(std::list<T> input)
{
    if(input.empty())
    {
        return input;
    }
    std::list<T> result;
    result.splice(result.begin(),input,input.begin());
    T const& pivot=*result.begin();
    auto divide_point=std::partition(input.begin(),input.end(),
            [&](T const& t){return t<pivot;});
    std::list<T> lower_part;
    lower_part.splice(lower_part.end(),input,input.begin(),
        divide_point);
    std::future<std::list<T> > new_lower(                           
        std::async(&parallel_quick_sort<T>,std::move(lower_part)));
    auto new_higher(
        parallel_quick_sort(std::move(input)));                     
    result.splice(result.end(),new_higher);                         
    result.splice(result.begin(),new_lower.get());                  
    return result;
}
  • A sample implementation of spawn task
template<typename F,typename A>
std::future<std::result_of<F(A&&)>::type>
    spawn_task(F&& f,A&& a)
{
    typedef std::result_of<F(A&&)>::type result_type;
    std::packaged_task<result_type(A&&)>
        task(std::move(f)));
    std::future<result_type> res(task.get_future());
    std::thread t(std::move(task),std::move(a));
    t.detach();
    return res;
}

Synchronizing operations with message passing.

  • The idea of csp is if there is no shared data each thread can reasoned about entirely independently, purely on the basis of how it behaves in response to the messages it received. Each thread is effectively a state machine.

    • The above can be achieved in two ways one method is formalise finite state machine mode.
    • The state machine can be implicit in the structure of the program.

  • True communicating sequential process have no shared data, with all the communications passed through message queues. But C++ threads share an address space, it is not possible to enforce this requirement.

Actor Model

  • There are several discrete actors in the system (each running on a separate thread). Which sends messages to each other to perform the task at hand and there is no shared state except that which is directly passed via messages.

Latches and Barriers

  • A latch is a synchronization object that becomes ready when its counter is reduced to zero.

  • Once its ready its stays ready until it is destroyed.

  • Latch is thus a light weight facility for waiting for a series of events to occur.

  • A Barrier is a reusable synchronization component used for internal synchronization between set of threads.

  • A Latch doesn't care which thread decrement the counter, that some thread can decrement the counter multiple times or multiple threads can decrement the counter once or some combination of two.

  • For Barrier each thread can only arrive at the barrier once per cycle.

A Basic latch type: std::latch

  • You specify the initial counter vale as one and only argument to the constructor.
  • The call to the count_down synchronises with the call to wait.
void foo(){
    unsigned const thread_count=...;
    latch done(thread_count);                                    
    my_data data[thread_count];
    std::vector<std::future<void> > threads;
    for(unsigned i=0;i<thread_count;++i)
        threads.push_back(std::async(std::launch::async,[&,i]{   
            data[i]=make_data(i);
            done.count_down();                                   
            do_more_stuff();                                     
        }));
    done.wait();                                                 
    process_data(data,thread_count);                            
}

std::barrier a basic barrier

  • Barriers release the waiting thread and then reset so they can be used again. Where as latch stay ready once they are.
  • Also barriers only synchronize with a group of threads. A thread cannot wait for a barrier to be ready unless it is one of the threads in the synchronization group.
result_chunk process(data_chunk);
std::vector<data_chunk>
divide_into_chunks(data_block data, unsigned num_threads);

void process_data(data_source &source, data_sink &sink) {
    unsigned const concurrency = std::thread::hardware_concurrency();
    unsigned const num_threads = (concurrency > 0) ? concurrency : 2;

    std::experimental::barrier sync(num_threads);
    std::vector<joining_thread> threads(num_threads);

    std::vector<data_chunk> chunks;
    result_block result;

    for (unsigned i = 0; i < num_threads; ++i) {
        threads[i] = joining_thread([&, i] {
            while (!source.done()) {                                   
                if (!i) {                                              
                    data_block current_block =
                        source.get_next_data_block();
                    chunks = divide_into_chunks(
                        current_block, num_threads);
                }
                sync.arrive_and_wait();                                
                result.set_chunk(i, num_threads, process(chunks[i]));  
                sync.arrive_and_wait();                                
                if (!i) {                                              
                    sink.write_data(std::move(result));
                }
            }
        });
    }
}
  • std::barrier has an additional constructor that takes a completion function as well as thread count.
  • The function will run on exactly one of the threads that arrived at the barrier, once all the threads have arrived at the barrier.
  • It provides a mean of changing the number of threads that must arrive at the barrier for next cycle.
  • The above example can written using completion function.
void process_data(data_source &source, data_sink &sink) {
    unsigned const concurrency = std::thread::hardware_concurrency();
    unsigned const num_threads = (concurrency > 0) ? concurrency : 2;

    std::vector<data_chunk> chunks;

    auto split_source = [&] {                                  
        if (!source.done()) {
            data_block current_block = source.get_next_data_block();
            chunks = divide_into_chunks(current_block, num_threads);
        }
    };

    split_source();                                            

    result_block result;

    std::experimental::flex_barrier sync(num_threads, [&] {    
        sink.write_data(std::move(result));
        split_source();                                        
        return -1;                                             
    });
    std::vector<joining_thread> threads(num_threads);

    for (unsigned i = 0; i < num_threads; ++i) {
        threads[i] = joining_thread([&, i] {
            while (!source.done()) {                           
                result.set_chunk(i, num_threads, process(chunks[i]));
                sync.arrive_and_wait();                        
            }
        });
    }
}
